Where does your code actually run?

The client/server mental model. You told me you already know this — it lives here as an optional refresher and to back the glossary. The course proper starts at Lesson 01 — Reading code.

Optional~5 minSkip if you know it

When the AI builds you a web app, it writes files that run in two completely different places. Almost every "wait, why doesn't this work / why is this insecure / where do I even change this?" moment comes from not knowing which place a piece of code lives in.

So before any syntax, learn the map. There are exactly two lands.

The two lands

The Client

Runs on the user's device — inside their web browser (Chrome, Safari…).

It's everything the user can see and touch: the buttons, the layout, the colors. Made of HTML, CSS, and JavaScript.

Also called: front‑end · client‑side · "the browser"

The Server

Runs on a computer you control — somewhere in the cloud, not on the user's device.

It does the private, trusted work: talking to the database, checking passwords, and calling AI APIs with your secret key.

Also called: back‑end · server‑side · "the backend"

They talk to each other by passing messages across the internet. The client sends a request ("hey, give me the dashboard data"); the server sends back a response ("here you go"). That back-and-forth is the heartbeat of every web app you'll ever ship. [MDN]

The rule that decides everything

Anything in the client, the user can see and tamper with. The browser hands them the full source — there are no secrets in the front-end. Anything you need to keep private or trusted has to live on the server.

Why this is the whole ballgame for an AI product

Here's the trap that catches almost every vibe coder shipping their first AI app. You get an API key from Anthropic or OpenAI. The fastest-looking thing is to paste it straight into the front-end JavaScript so the browser can call the AI. It works in your demo — so it feels right.

But remember the rule: the browser shows users everything. Anyone who opens your site can read that key, and now they're running up your bill on your dime. This isn't rare; it's one of the most common ways new AI apps get drained.

The fix falls right out of the mental model: the browser asks your server for an answer → your server (where secrets are safe) calls the AI → your server passes the result back. The key never leaves the trusted land.

// ❌ DANGER — runs in the browser, key is visible to everyone
fetch("https://api.anthropic.com/...", {
  headers: { "x-api-key": "sk-ant-my-secret-key" }  // 😱 exposed
})

// ✅ SAFE — browser talks to YOUR server; the key stays server-side
fetch("/api/ask-ai", { method: "POST", body: question })
// ...your server holds the key and calls the AI on the browser's behalf.

You don't need to write this yet. The win is this: when the AI generates code like the top snippet, you can now spot it — because you know which land the key is standing in.
