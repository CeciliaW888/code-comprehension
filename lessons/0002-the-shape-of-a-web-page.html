Lesson 02 · The Shape of a Web Page

A web page is a living tree, not a document.

Why "View Source" never matches what's on your screen — and what React is actually doing the moment the AI reaches for it.

Mental model~5 minMission: read the frontend

You can already tweak a bit of HTML. So here's the puzzle that HTML alone can't explain: your AI chatbot starts with an empty page, then messages appear as you talk to it. But if you right-click and hit "View Source," none of those messages are in there. Where did they come from, and why isn't the source telling the truth?

The answer is the single idea that makes every frontend — plain HTML or a big React app — stop being a mystery.

The rule that unlocks the frontend

HTML is only the starting shape. The browser reads it once and builds a living tree of objects called the DOM. From then on, the page you see is that tree — and JavaScript's whole job is to change it: add branches, remove them, edit their text.

The three roles (you know two of these)

Every frontend is three jobs working together. Name the role and you know which file to open.

HTML · structure

The content and its shape — headings, buttons, the chat box. This is the tree's starting form.

CSS · appearance

The colors, spacing, fonts. It styles the tree but never changes what's in it.

JavaScript · behavior & change

The only one of the three that can reach into the tree and rewrite it while the page is open — adding a new message, hiding a menu, updating a counter. When a page does something after it loads, JavaScript did it.

See the tree

Here's a scrap of HTML and the tree the browser builds from it. Nesting in the HTML becomes parent/child branches in the tree — that's all "the DOM" means.

<!-- the HTML you wrote -->
<div class="chat">
  <p>Hello</p>
</div>

// the tree (DOM) the browser builds from it:
//   div.chat
//     └─ p   →  "Hello"

Now JavaScript adds a second message. It doesn't rewrite your HTML file — it hangs a new branch on the live tree:

// JS reaches into the tree and adds a branch
let msg = document.createElement("p");
msg.textContent = "How can I help?";
document.querySelector(".chat").appendChild(msg);

// the tree is now:
//   div.chat
//     ├─ p  →  "Hello"
//     └─ p  →  "How can I help?"   ← new, never in View Source

That's the whole mystery solved. View Source shows the original HTML the server first sent. The screen shows the current tree — which JavaScript has been editing ever since. To see the real, live tree instead, you use Inspect (we'll do that below).

So what is React, really?

Editing the tree by hand — createElement, appendChild, find this, remove that — gets painful fast in a real app. That pain is the entire reason React exists.

React in one sentence

Instead of you writing step-by-step tree edits, you describe what the tree should look like for a given piece of data — and React makes the real DOM match it. Your chat messages live in a list (data); React keeps the tree in sync with that list, so when you add a message, the branch appears. You stopped giving directions and started describing the destination.

This is why React code looks like HTML mixed into JavaScript (that's JSX): you're not writing the page, you're writing a description of the tree that React will build and keep updated. When you trace React code now, read it as: "what tree does this data produce?"

Your one tangible win

Open any web app the AI built you. Right-click anything on the page → Inspect. That panel is the live DOM tree — not View Source. Now do something in the app (send a message, open a menu) and watch a branch appear or change in real time. Seeing the tree move is the moment the frontend stops being magic.
