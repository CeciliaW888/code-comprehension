Lesson 01 · Reading Code With Confidence

Stop reading code like prose. Run it in your head.

This is your actual bottleneck: AI-generated code runs, but you can't say what it does or where to change it. The fix is one habit — and it's learnable in an afternoon.

Skill~5 minMission: read & modify the code

You already know where code runs — client vs. server (there's an optional primer if you ever want the refresher). Now the harder thing: reading a chunk and knowing what it does. The reason it feels opaque isn't that the code is hard — it's that you've been reading it the way you read a paragraph: left to right, grabbing the gist.

Code doesn't have a "gist." A computer doesn't skim — it executes one line at a time, and each line changes the state of something. To read code, you do the same thing the computer does.

The rule that unlocks reading

Be the computer. Go line by line, top to bottom, and keep a running list of what each variable currently holds. Don't ask "what does this mean?" — ask "what value is in each box right now, and what does this line change?"

The only three things any line is doing

Ninety percent of code you'll ever read is built from three moves. Once you can name which one you're looking at, the wall comes down.

1 · Variables

A labeled box holding a current value. let limit = 5 puts 5 in the box called limit. The value can change later — that's the whole point of tracing.

2 · Functions

A recipe: inputs go in, one result comes out. To read one, find the inputs (the stuff in parentheses) and the output (the return). Ignore the middle until you need it.

3 · Control flow — the forks

if picks which lines run; a loop (for, while) runs the same lines again and again. These are the only places the path through the code can branch or repeat. Everything else just runs straight down.

Watch it work — a real AI-product function

Here's the kind of thing the AI writes for a chatbot: take the conversation so far, keep only the recent messages, and stitch them into one block of text to send to the model.

function buildReply(messages, isPro) {
  let limit  = isPro ? 20 : 5;     // how many past messages to keep
  let recent = messages.slice(-limit);  // the last `limit` of them
  let text   = "";
  for (let m of recent) {
    text += m.role + ": " + m.content + "\n";
  }
  return text.trim();
}

Don't skim it. Trace it with a concrete call — a free user (isPro is false) with two messages. Keep the box-table in your head as you step down:

// the call we're tracing:
buildReply([{role:"user", content:"hi"},
            {role:"ai",   content:"hello"}], false)

// line by line — what's in each box:
// limit  -> false ? 20 : 5     so 5
// recent -> last 5 of 2 items  so both messages
// text   -> ""                 then the loop runs twice:
//   pass 1:  text = "user: hi\n"
//   pass 2:  text = "user: hi\nai: hello\n"
// return -> text.trim()  =>  "user: hi\nai: hello"

That's it. You didn't memorize syntax — you tracked values down the page and arrived at the answer the computer would. The ? chose a number, the loop built up a string, the return handed it back. Reading is tracing.

Why this is the whole mission

Once you can trace, "I can't modify it" dissolves too. The user says the bot forgets too fast? You don't hunt blindly — you know the memory length lives in one box, limit. Tracing tells you exactly where to put your hands.
